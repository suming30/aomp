# AOM Codex Thread Guide

## Recommended Thread Groups

### `00-项目总控`

- 用途：记录当前版本目标、阶段进度、全局待办和优先级。
- 适合提问：这个版本先做什么、目前还差哪些模块、现在的风险点是什么。
- 示例指令：
  - “先根据当前仓库和文档，帮我总结一下这个版本的主线任务。”
  - “把接下来的工作拆成前端、后端、测试三个部分。”

### `01-产品与需求`

- 用途：讨论 PRD、功能边界、页面流程、权限规则和验收口径。
- 适合提问：这个功能怎么拆、某个角色能不能做某个动作、验收标准怎么写。
- 示例指令：
  - “先不要写代码，帮我把这个功能整理成需求和验收标准。”
  - “根据 `docs/product` 里的文档，提炼本期要做的功能点。”

### `02-前端开发`

- 用途：专门处理 `aomp-admin` 相关工作。
- 适合提问：页面改造、组件开发、接口接入、状态管理、前端报错修复。
- 示例指令：
  - “扫描 `aomp-admin`，找出和登录页面有关的文件并直接修改。”
  - “不要只解释，直接在前端里实现这个需求并告诉我改了哪些文件。”

### `03-后端开发`

- 用途：专门处理 `aomp-server` 相关工作。
- 适合提问：接口设计、业务逻辑、权限处理、后端测试、启动报错。
- 示例指令：
  - “扫描 `aomp-server`，定位用户权限校验逻辑。”
  - “直接修复这个接口的异常，并说明影响范围。”

### `04-联调与测试`

- 用途：集中处理联调问题、测试用例、截图、测试报告和验收结论。
- 适合提问：接口联调失败、回归检查、测试覆盖、报告整理。
- 示例指令：
  - “根据 `docs/testing` 和 `artifacts` 现有材料，整理当前联调状态。”
  - “帮我生成一版本轮测试结论，缺陷和通过项分开写。”

## Suggested Workflow

1. 先在 `01-产品与需求` 线程把目标、边界和验收标准聊清楚。
2. 再进入 `02-前端开发` 或 `03-后端开发` 线程执行实际改动。
3. 如果遇到联调或验证问题，切到 `04-联调与测试` 线程集中处理。
4. 需要全局排期、优先级调整或阶段总结时，再回到 `00-项目总控`。

## Practical Rules

- 讨论需求时尽量不要和改代码混在同一个线程里。
- 真正要我查文件、改代码、跑命令时，尽量在绑定仓库目录的线程里进行。
- 提需求时最好同时说明目标、范围、限制和验收标准。
- 提 bug 时最好同时附上报错信息、复现步骤和你怀疑的模块。
- 每个线程尽量长期保持单一主题，后续回看会非常清晰。
